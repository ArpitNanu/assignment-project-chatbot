# AI Document Chat Assistant - Backend Architecture

This document breaks down the backend implementation of the AI Document Chat Assistant. It is designed to help you understand the architecture and explain it clearly to an interviewer.

## 1. Core Architecture

The backend follows the **MVC (Model-View-Controller)** pattern, specifically adapted for a REST API where the "View" is replaced by JSON responses sent to the frontend.

We chose this modular structure to separate concerns:
*   **Models**: Define the database structure.
*   **Routes**: Define the API endpoints (URLs).
*   **Middleware**: Handle logic that runs *before* the route (like checking if a user is logged in, or handling file uploads).
*   **Controllers**: The actual logic is currently kept inside the route files to keep the project simple and easy to explain.

## 2. File-by-File Breakdown

### The Root
*   **`server.js`**: This is the entry point of the application. It initializes the Express app, connects to MongoDB, sets up global middleware (like `cors` and `express.json`), and registers all the API routes (`/api/auth`, `/api/documents`, `/api/chat`).
*   **`.env`**: Stores sensitive variables like `MONGO_URI`, `JWT_SECRET`, and `GEMINI_API_KEY`. This file is never pushed to GitHub.

### `/models` (Database Schemas)
*   **`User.js`**: Defines the user schema (name, email, password).
    *   **How it works**: It uses Mongoose middleware (`pre('save')`) to automatically hash passwords using `bcryptjs` before they are saved to the database. It also includes a custom method (`matchPassword`) to verify passwords during login.
*   **`Document.js`**: Stores the uploaded documents.
    *   **How it works**: Instead of just storing a file path, we extract the text from the PDF and store the raw string in the `content` field. It is linked to a specific user via `userId`.
*   **`Chat.js`**: Stores the conversation history.
    *   **How it works**: Contains the `question` asked by the user, the `answer` generated by Gemini, and the `userId` to ensure privacy.

### `/middleware` (Interceptors)
*   **`authMiddleware.js` (`protect`)**: 
    *   **How it works**: Intercepts requests to private routes. It looks for a `Bearer Token` in the Authorization header. It uses `jsonwebtoken` to verify the token. If valid, it attaches the `req.user` to the request, allowing the route to proceed. If invalid, it blocks the request with a `401 Unauthorized`.
*   **`uploadMiddleware.js` (`upload`)**:
    *   **How it works**: Uses `multer` to handle `multipart/form-data` (file uploads). It is configured to use `memoryStorage()`, meaning the uploaded file is kept temporarily in RAM as a Buffer, rather than being saved to the hard drive.

### `/routes` (API Endpoints)
*   **`authRoutes.js`**: Handles `/signup` and `/login`.
    *   **How it works**: Validates credentials and uses `jwt.sign()` to generate a token containing the user's ID.
*   **`documentRoutes.js`**: Handles `/upload`.
    *   **How it works**: It takes the uploaded file buffer from Multer, passes it to `pdf-parse` to extract the raw text, and then saves that text to the `Document` collection alongside the user's ID.
*   **`chatRoutes.js`**: Handles the AI logic.
    *   **How it works**: This implements a basic **RAG (Retrieval-Augmented Generation)** flow. It receives a `documentId` and a `question`. It fetches the document from MongoDB, injects the document's text into a strict prompt template, and sends it to the Gemini API. Finally, it saves the interaction to the `Chat` collection.

## 3. Potential Interview Questions & Answers

### Q1: Why did you use `memoryStorage` in Multer instead of `diskStorage`?
**Answer:** "Since our main goal is to extract the text from the PDF and pass it to the AI, we don't actually need to keep the physical PDF file permanently. By using `memoryStorage`, the file is kept in RAM as a buffer just long enough for `pdf-parse` to read the text. This saves disk space on the server and makes the operation slightly faster because we avoid disk I/O."

### Q2: How does the AI know about the user's document? What is the concept behind this?
**Answer:** "I implemented a pattern called **Retrieval-Augmented Generation (RAG)**. Large Language Models like Gemini don't inherently know about a user's private PDF. So, when the user asks a question, my backend *retrieves* the text of the PDF from our database, and *augments* the prompt with that text as context before passing it to Gemini. I explicitly instruct the model in the prompt to *only* use the provided context to answer the question, which prevents the AI from hallucinating."

### Q3: How are you securing user passwords?
**Answer:** "I never store passwords in plain text. I use `bcryptjs` inside a Mongoose `pre-save` hook. When a user is created, the password is salted (with a salt factor of 10) and hashed before saving. During login, I use bcrypt's `compare` function to check if the entered password matches the stored hash."

### Q4: Explain your Authentication flow. How do you protect routes?
**Answer:** "The application uses **JWT (JSON Web Tokens)** for stateless authentication. When a user logs in successfully, the server generates a JWT containing the user's ID and signs it with a secret key. The frontend must send this token in the `Authorization` header for subsequent requests. My backend has an `authMiddleware` that intercepts requests, verifies the token's signature, and attaches the user object to the request. If the token is missing or invalid, it returns a 401 error."

### Q5: Why did you choose Gemini 1.5 Flash over other models?
**Answer:** "Gemini 1.5 Flash is highly optimized for speed and cost-efficiency while still supporting a massive context window. Since we are passing entire document contents (which can be thousands of words) into the prompt, we need a model that can process large amounts of context quickly without causing the user to wait a long time for a chat response."

### Q6: How do you ensure that User A cannot see User B's documents or chats?
**Answer:** "Both the `Document` and `Chat` schemas have a required `userId` reference field. In the protected routes, I extract the `req.user._id` from the verified JWT token. Whenever a document or chat is saved, it is hardcoded to that user ID. When fetching history, the database query strictly filters by `userId: req.user._id`, making it impossible to access another user's data."
